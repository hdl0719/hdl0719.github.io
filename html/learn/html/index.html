使用 web Workers 提升页面性能

  1. 原理
    • 独立线程：Web Workers 在主线程之外运行，不会阻塞 UI 渲染和用户交互。
    • 通信机制：通过 postMessage 和 onmessage 与主线程通信，数据通过结构化克隆算法传递（支持大多数数据类型，但无法传递函数或 DOM 元素）。
  2. 使用场景
    • 处理 CPU 密集型任务
    • 处理网络请求
    • 并行处理
    • 消息批处理
  3. 缺点：
    • 无 DOM 访问：Worker 中无法操作 DOM 或调用 window API。
    • 同源限制：Worker 脚本需与主线程同源（可通过 Blob URL 绕过）。
    • 调试工具：浏览器开发者工具支持 Worker 调试（如 Chrome 的 Sources 面板）。
    • 浏览器限制

localStorage 和 sessionstorage 什么时候会消失，怎样清除

SessionStorage 失效时间

sessionStorage 的生命周期仅在当前会话有效。sessionStorage 引入了“浏览器窗口的概念”。sessionStorage 在同源窗口中始终存在数据，只要浏览器窗口没有关闭，刷新或者重新进入页面数据依然存在。关闭浏览器窗口后数据会被删除。再次独立打开同一个窗口同一个页面，sessionStorage 也是不一样。

SessionStorage 存储的位置

sessionStorage 都保存在客户端，一般不与服务器进行通信交互。

SessionStorage 存储的大小

sessionStorage 存储数据大小一般是 5MB

localStorage 的失效时间

localStorage 的生命周期是永久的，关闭页面或者浏览器之后 localStorage 中的数据也不会消失。localStorage 删除数据要手动删除，否则数据永远不会消失。

调用 localStorage.clear()

localStorage 存储的位置

localStorage 都保存在客户端，一般不与服务器进行通信交互。

localStorage 存储的大小

localStorage 存储数据大小一般是 5MB

1.相同点

  • 纯浏览器端存储, 存储空间在 2.5M 到 10M 之间（各浏览器不同）, 请求时不会自动携带
  • 只能保存文本, 如果是对象或数组, 需要转换为 JSON

2.不同点

  • localStorage 保存在本地文件中, 除非编码或手动删除, 否则一直存在
  • sessonStorage 数据保存在当前会话内存中, 关闭浏览器则清除

HTTP 包含哪些内容

  • 请求头：元信息、Host（主机名）、Content-Type、User-Agent（浏览器版本）
  • 请求行：请求方式（POST、GET 等），URL 路径，HTTP 版本
  • 请求体：请求参数等
  • 响应头：Content-Type，Content-length（响应内容长度字节）
  • 响应行：HTTP 协议版本，响应状态码，状态码描述—200 成功，404 请求资源不存在，500 服务端错误
  • 响应体：响应内容

TCP 和 HTTP、UDP 区别

  • TCP (传输控制协议，Transmission Control Protocol) ，(类似打电话)面向连接、传输可靠(保证数据正确性) 、有序(保证数据顺序) 、传输大量数据 (流模式) 、速度慢、对系统资源的要求多，程序结构较复杂

    每一条 TCP 连接只能是点到点的，TCP 首部开销 20 字节。

  • UDP(用户数据报协议，User Data Protocol) : (类似发短信)面向非连接、传输不可靠 (可能丢包)、无序、传输少量数据(数据报模式)、速度快，对系统资源的要求少，程序结构较简单。

    UDP 支持一对一，一对多，多对一和多对多的交互通信，UDP 的首部开销小，只有 8 个字节。

    • TCP 和 UDP:传输层协议， (类似于卡车)

    • HTTP:应用层协议， (类似于货物) 。

TCP 确保数据可靠性和三次握手

TCP 建立连接需要三次握手

  • • 第一次握手:客户端发送 SYN 包(seg=x) 到服务器，并进入 SYN SEND 状态，等待服务器确认;
  • • 第二次握手:服务器收到 SYN 包，必须确认客户的 SYN (ack=x+1) ，同时自己也发送一个 SYN 包 (seg=y) ，即 SYN+ACK 包，此时服务器进入 SYN RECV 状态
  • • 第三次握手: 客户端收到服务器的 SYN+ACK 包，向服务器发送确认包 ACK(ack=y+1)，此包发送完毕，客户端和服务器进入 ESTABLISHED 状态，完成三次握手

握手过程中传送的包里不包合数据，三次握手完毕后，客户端与服务器才正式开始传送数据。

react 页面白屏检测

  • 资源加载失败页面依赖的关键资源(CSS、JS、图片等)加载失败,导致页面无法正常渲染

    这种情况下我们可以在关键的接口和请求上增加响应监控。

  • 资源加载延迟资源加载延迟(或阻塞)，导致页面长时间等待资源加载完成

    这种情况可以前端做一些loading状态或者增加骨架屏，防止页面无内容造成用户困扰

  • 代码执行中出现未被捕捉的错误，例如 JavaScript 执行错误，Promise 错误等等。导致页面功能无法正常工作，出现空白。

  • 浏览器兼容问题

  1. 全局错误监听 + 判断对应节点是否存在

    首先，创建一个名为 ErrorBoundary 的新组件。在这个组件中，我们需要定义一个名为 componentDidCatch 的生命周期方法，它将在子组件中捕获到错误时被调用。同时，我们需要在组件的状态中存储一个表示是否发生错误的变量。

    接下来，将 ErrorBoundary 组件包裹在应用程序的根组件中，以便捕获整个应用程序的错误。

重排重绘

重绘：一个元素的外观的改变会触发浏览器行为即样式改变会引起重绘，「重绘是以图层为单位的，如果图层中某个元素需要重绘，那么整个图层都需要重绘」。所以为了提高性能，我们应该**「让那些”变化的元素”单独作为一个图层，将其隔离起来」**

2、重排：页面中元素的布局位置就是重排。「重绘不一定会导致重排，重排会一定导致重绘。」

3、「一个简单的 html 页面来谈浏览器的渲染流程：DOM 解析生成 DOM 树–>CSS 解析生成 CSS 树–>样式计算–>布局树–>图层树–>绘制（GUP 线程处理）–>合并图层」

4、图层有：css 3d、css3 动画、position:fixed、will-change：transform 作为单独的图层、video、canvas

5、减少重绘重排的方案：transform 来代替 top、left 操作、使用 opacity 来替代 visibility、将多次改变样式的操作合并成一次、添加元素使用文档碎片（documentFragment)、不要在循环中获取 DOM 节点的属性（with、height 会引发重绘重排）、为动画元素新建图层

treeShaking 有什么前置条件 是用什么场景

前置条件

  1. 使用 ES Modules (ESM) 语法
    • Tree Shaking 依赖静态的 import/export 语法（ES6 模块化），动态导入（如 require）无法被静态分析。
  2. 模块标记为 sideEffect-free
    • 在 package.json 中设置 "sideEffects": false，或明确声明有副作用的文件（如 CSS、全局样式等）。
    • 若模块有副作用（如修改全局变量、立即执行的代码），需显式声明以避免被误删。
  3. 生产环境构建
    • Tree Shaking 通常在 mode: 'production'（Webpack）或生产模式构建时生效，开发模式下可能不启用。
  4. 打包工具支持
    • Webpack 4+、Rollup、Vite 等工具原生支持，需确保配置正确（如 Webpack 的 optimization.usedExports: true）

适用场景

  1. 优化第三方库体积

    • 例如 Lodash 的按需引入：

  2. 项目代码按需加载

    • 移除未使用的组件、工具函数等，减少最终打包体积。

  3. Vue/React 等框架的优化

    • Vue 3 的 Composition API 和 React 的函数组件更易被 Tree Shaking 优化。

  4. 配合代码分割（Code Splitting）

浏览器渲染的过程

  1. 解析 HTML 构建 DOM 树：
    • 浏览器将 HTML 文档解析成 DOM（文档对象模型）树结构。
    • 遇到 CSS 和 JS 时，CSS 解析不会阻塞 HTML 解析，但 JS 会阻塞（除非标记为async或defer）。
  2. 解析 CSS 构建 CSSOM 树：
    • 解析 CSS 样式表生成 CSSOM（CSS 对象模型）树，确定每个节点的样式规则。
  3. 构建渲染树（Render Tree）：
    • 结合 DOM 树和 CSSOM 树，生成渲染树，只包含可见的节点（如display: none的元素不会包含在内）。
  4. 布局（Layout/Reflow）：
    • 计算每个渲染树节点的几何信息（位置、大小等），生成布局树。
  5. 分层（Layer）：
    • 根据层叠上下文、transform、opacity等属性将页面分为多个图层，提升渲染效率。
  6. 绘制（Paint）：
    • 为每个图层生成绘制指令列表，描述如何绘制每个元素（如颜色、边框等）。
  7. 分块与光栅化（Tiling & Rasterization）：
    • 合成线程将图层分割为小块（tiles），并交给 GPU 进程光栅化为位图。
  8. 合成与显示（Composite & Display）：
    • 合成线程将光栅化后的位图合成为最终画面，通过 GPU 绘制到屏幕上。

关键优化点：

  • 减少重排（Reflow）和重绘（Repaint）：避免频繁修改几何属性（如width、height），优先使用transform和opacity（仅触发合成阶段）。
  • 使用will-change：提示浏览器提前优化可能变化的元素。
  • 避免强制同步布局：如连续读取offsetWidth后再修改样式，会导致多次重排。

一、什么是服务端渲染（SSR）,和客户端渲染（CSR）有什么不一样，优缺点？SSR 是一种将网页内容在服务器端生成并发送到客户端的技术。与客户端渲染（Client-Side Rendering，CSR）不同，SSR 是在服务器上完成页面的渲染工作，然后将完整的 HTML 页面发送给浏览器。 服务端渲染的优点 更快的首屏加载速度用户无需等待 JavaScript 下载和执行，可以直接看到完整的页面内容。 更好的 SEO（搜索引擎优化）搜索引擎可以直接抓取服务器生成的 HTML 内容，有利于提高搜索排名。 更好的用户体验对于网络较慢或设备性能较差的用户，SSR 可以提供更快的页面加载体验。 兼容性更好不依赖浏览器的 JavaScript 执行能力，适合老旧设备或低版本浏览器。 服务端渲染的缺点 服务器压力大每次请求都需要服务器渲染页面，增加了服务器的负载。 开发复杂度高需要处理服务器和客户端的代码兼容性，增加了开发难度。 页面切换可能较慢页面切换时，可能需要重新请求服务器，导致体验不如 CSR...
