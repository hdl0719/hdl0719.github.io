一、什么是服务端渲染（SSR）,和客户端渲染（CSR）有什么不一样，优缺点？

SSR 是一种将网页内容在服务器端生成并发送到客户端的技术。与客户端渲染（Client-Side Rendering，CSR）不同，SSR 是在服务器上完成页面的渲染工作，然后将完整的 HTML 页面发送给浏览器。

服务端渲染的优点

  1. 更快的首屏加载速度
    用户无需等待 JavaScript 下载和执行，可以直接看到完整的页面内容。
  2. 更好的 SEO（搜索引擎优化）
    搜索引擎可以直接抓取服务器生成的 HTML 内容，有利于提高搜索排名。
  3. 更好的用户体验
    对于网络较慢或设备性能较差的用户，SSR 可以提供更快的页面加载体验。
  4. 兼容性更好
    不依赖浏览器的 JavaScript 执行能力，适合老旧设备或低版本浏览器。

服务端渲染的缺点

  1. 服务器压力大
    每次请求都需要服务器渲染页面，增加了服务器的负载。
  2. 开发复杂度高
    需要处理服务器和客户端的代码兼容性，增加了开发难度。
  3. 页面切换可能较慢
    页面切换时，可能需要重新请求服务器，导致体验不如 CSR 流畅。

服务端渲染的适用场景

  • 内容型网站：如新闻、博客、电商商品详情页等，需要快速加载和 SEO 优化。
  • 静态页面：页面内容变化较少，适合预渲染。
  • 对 SEO 要求高的应用：如企业官网、营销页面等。

二、SSR 的第三方库问题

1、浏览器 API 的缺失

​ 条件渲染：在组件中判断当前环境是否为浏览器，避免在服务器端调用浏览器 API。

​ 动态导入：使用动态导入（Dynamic Import）在客户端加载依赖浏览器 API 的库。

2、 CSS 和样式问题

​ 提取样式：使用支持 SSR 的 CSS-in-JS 库（如 styled-components 或 emotion），并确保在服务器端提取样式。

​ 预加载样式：在 HTML 中预加载关键 CSS，避免样式闪烁。

3、异步数据加载

​ 预取数据：在服务器端预取数据，并将数据注入到页面中。

​ 避免客户端重复加载：确保客户端不会重复加载已经在服务器端获取的数据。

4、全局变量污染

​ 隔离环境：确保服务器端和客户端的全局变量相互隔离。

​ 清理全局变量：在每次请求后清理全局变量，避免污染其他请求。

5、生命周期钩子问题

​ 条件执行：在生命周期钩子中判断当前环境是否为浏览器。

​ 使用 useEffect：在函数组件中使用 useEffect，它只会在客户端执行。

6、路由问题

​ 静态路由配置：在服务器端使用静态路由配置。

​ 匹配路由：使用 react-router 的 StaticRouter 在服务器端匹配路由。

7、文件大小和性能问题

​ 代码分割：使用代码分割（Code Splitting）减少初始加载的代码量。

​ 按需加载：仅在需要时加载第三方库。

8、环境变量问题

​ 统一环境变量：使用构建工具（如 Webpack）将环境变量注入到客户端和服务器端。

​ 动态配置：在运行时动态加载配置。

三、SSR 中进行数据处理有什么不同吗？比如正则或者 symbol

1. 正则表达式（RegExp）

问题描述

正则表达式在 SSR 和 CSR 中的行为通常是一致的，但在某些情况下可能会遇到问题：

  • 环境差异：某些正则表达式可能依赖于浏览器特有的行为（如 window 对象的属性）。
  • 性能问题：复杂的正则表达式在服务器端可能会影响性能，尤其是在高并发场景下。

解决方法

  • 避免浏览器依赖：确保正则表达式不依赖于浏览器环境。
  • 预编译正则表达式：在服务器端预编译正则表达式，避免重复编译。
  • 性能优化：对于复杂的正则表达式，考虑使用更高效的算法或工具（如 Intl API）。

2. Symbol

问题描述

Symbol 是 JavaScript 中的一种基本数据类型，用于创建唯一值。在 SSR 中，Symbol 的使用可能会遇到以下问题：

  • 序列化问题：Symbol 无法被序列化为 JSON，因此在 SSR 中传递数据时需要注意。
  • 环境一致性：Symbol 在服务器端和客户端必须是唯一的，否则可能会导致不一致。

解决方法

  • 避免传递 Symbol：在 SSR 中尽量避免将 Symbol 作为数据传递给客户端。
  • 使用字符串代替：如果需要唯一标识符，可以使用字符串代替 Symbol。
  • 确保一致性：如果必须在 SSR 和 CSR 中使用 Symbol，确保其定义一致。

3. 数据处理的其他注意事项

(1) 数据序列化

  • 问题：在 SSR 中，数据需要从服务器传递到客户端，因此必须能够被序列化为 JSON。
  • 解决方法：避免传递不可序列化的数据（如函数、Symbol、RegExp 等）。

(2) 异步数据加载

  • 问题：SSR 需要在渲染页面之前完成所有数据加载，而 CSR 可以在页面加载后异步获取数据。
  • 解决方法：使用 getServerSideProps（Next.js）或类似机制在服务器端预取数据。

(3) 环境差异

  • 问题：某些数据处理逻辑可能依赖于浏览器环境（如 window、document）。
  • 解决方法：通过环境判断（typeof window === 'undefined'）隔离服务器端和客户端的逻辑。

(4) 性能问题

  • 问题：SSR 中数据处理可能会增加服务器负载，尤其是在高并发场景下。
  • 解决方法：优化数据处理逻辑，使用缓存（如 Redis）减少重复计算。

四、性能指标及优化

1. 性能指标

(1) 加载性能

  • 首次内容绘制（First Contentful Paint, FCP）
    测量页面首次渲染任何文本、图像或非空白内容的时间。
  • 最大内容绘制（Largest Contentful Paint, LCP）
    测量页面中最大内容元素（如图片、视频或文本块）的渲染时间。

(2) 交互性能

  • 首次输入延迟（First Input Delay, FID）
    测量用户首次与页面交互（如点击按钮）到浏览器实际响应的延迟时间。
  • 总阻塞时间（Total Blocking Time, TBT）
    测量页面在 FCP 和 TTI 之间主线程被阻塞的总时间。

(3) 视觉稳定性

  • 累积布局偏移（Cumulative Layout Shift, CLS）
    测量页面加载期间元素的意外移动程度。

2. 性能优化策略

根据性能指标和分析结果，可以采取以下优化策略：

(1) 减少加载时间

  • 代码分割：使用动态导入（Dynamic Import）分割代码，减少初始加载体积。
  • 压缩资源：压缩 JavaScript、CSS、图片等资源。
  • 缓存策略：使用 HTTP 缓存（如 Cache-Control）减少重复请求。

(2) 提高渲染性能

  • SSR 优化：在服务器端预渲染页面，减少客户端渲染压力。
  • 懒加载：延迟加载非关键资源（如图片、视频）。
  • 优化 CSS：避免复杂的 CSS 选择器，减少重绘和回流。

(3) 减少交互延迟

  • 优化 JavaScript：减少主线程阻塞，使用 Web Workers 处理复杂计算。
  • 预加载关键资源：使用 <link rel="preload"> 提前加载关键资源。

(4) 提高视觉稳定性

  • 预留空间：为动态加载的内容预留空间，避免布局偏移。
  • 优化字体加载：使用 font-display: swap 避免字体加载时的布局偏移。

五、预渲染的内容

预渲染的内容通常包括以下部分：

1. HTML 结构

  • 页面骨架：预渲染会生成页面的完整 HTML 结构，包括 <head> 和 <body> 部分。
  • 静态内容：页面中的静态内容（如文本、图片、链接等）会被预先渲染。

2. CSS 样式

  • 内联样式：部分 CSS 可能会被内联到 HTML 中，以减少首次渲染时的请求。
  • 关键 CSS：提取关键路径 CSS（Critical CSS），确保首屏内容快速渲染。

3. JavaScript 数据

  • 初始状态：如果页面依赖于 JavaScript 数据（如 React、Vue 的状态），预渲染可以将初始状态嵌入到 HTML 中。
  • 占位符内容：对于动态内容，可以使用占位符（如加载动画）提前渲染。

4. 元数据（Meta Tags）

  • SEO 元数据：包括标题（<title>）、描述（<meta description>）、开放图谱标签（Open Graph Tags）等。
  • 结构化数据：如 Schema.org 标记，帮助搜索引擎理解页面内容。

5. 图片和媒体

  • 图片占位符：使用低分辨率图片或 Base64 编码的占位符，确保图片加载时页面布局稳定。
  • 懒加载媒体：对于非首屏的图片或视频，可以使用懒加载技术。

六、为什么微前端不适用 iframe

iframe 最大的特性就是提供了浏览器原生的硬隔离方案，不论是样式隔离、js 隔离这类问题统统都能被完美解决。但他的最大问题也在于他的隔离性无法被突破，导致应用间上下文无法被共享，随之带来的开发体验、产品体验的问题。

  1. url 不同步。浏览器刷新 iframe url 状态丢失、后退前进按钮无法使用。
  2. UI 不同步，DOM 结构不共享。想象一下屏幕右下角 1/4 的 iframe 里来一个带遮罩层的弹框，同时我们要求这个弹框要浏览器居中显示，还要浏览器 resize 时自动居中..
  3. 全局上下文完全隔离，内存变量不共享。iframe 内外系统的通信、数据同步等需求，主应用的 cookie 要透传到根域名都不同的子应用中实现免登效果。
  4. 慢。每次子应用进入都是一次浏览器上下文重建、资源重新加载的过程。

如何防止 XSS 和 CSRF 攻击

1. 防止 XSS（跨站脚本攻击）

XSS 攻击是指攻击者在网页中注入恶意脚本，当其他用户访问该页面时，脚本会在其浏览器中执行。

防护措施：

(1) 输入过滤与验证

  • 对用户输入的数据进行严格的过滤，移除或转义特殊字符（如 <, >, ", ', &）。
  • 使用白名单机制，只允许特定的 HTML 标签和属性（如富文本编辑器场景）。

(2) 输出编码

  • 在将用户输入的内容输出到 HTML、JavaScript、CSS 或 URL 时，进行适当的编码：
    • HTML 实体编码（如 < → <）
    • JavaScript 编码（如 " → \x22）
    • URL 编码（如 & → %26）

(3) 使用 Content Security Policy (CSP)

  • 通过 HTTP 头 Content-Security-Policy 限制脚本来源，防止内联脚本和外部恶意脚本执行：

(4) 设置 HttpOnly 和 Secure Cookie

  • 防止 JavaScript 通过 document.cookie 读取敏感 Cookie：

(5) 避免使用 innerHTML，改用 textContent

  • 直接操作 DOM 时，避免使用 innerHTML，改用更安全的 API（如 textContent）。

2. 防止 CSRF（跨站请求伪造）

CSRF 攻击是指攻击者诱导用户访问恶意网站，并在用户已登录目标网站的情况下，伪造用户发起非预期的请求（如转账、修改密码等）。

防护措施：

(1) 使用 CSRF Token

  • 服务器生成随机 Token 并嵌入表单或 AJAX 请求中，提交时验证 Token 是否匹配：

(2) 检查 Referer 和 Origin 头

  • 验证请求的来源是否合法：

(3) 设置 SameSite Cookie

  • 限制 Cookie 仅在同站点请求时发送，防止跨站请求携带 Cookie：

    • SameSite=Strict：完全禁止跨站携带 Cookie。
    • SameSite=Lax（默认）：允许部分安全请求（如 GET）跨站携带 Cookie。

(4) 关键操作要求二次验证

  • 敏感操作（如支付、修改密码）要求用户输入密码或验证码。

(5) 避免使用 GET 请求修改数据

  • RESTful API 设计时，遵循 GET 只用于读取数据，POST/PUT/DELETE 用于修改数据。

3. 其他通用安全措施

  • 定期更新框架和库（如 Django、React 等），避免已知漏洞。
  • 使用 HTTPS 防止中间人攻击（MITM）。
  • 限制用户权限（如最小权限原则）。
  • 日志监控，及时发现异常请求。

总结对比表

通过综合使用这些方法，可以大幅降低 XSS 和 CSRF 攻击的风险。

使用 websocket 实现数据通信

1.WebSocket 基础

特点：

•全双工通信：客户端和服务器可以同时发送和接收数据，而不像 HTTP 需要请求-响应模式。

•减少带宽消耗：相比 HTTP 轮询，WebSocket 只在建立连接时进行一次 HTTP 请求，之后使用单个 TCP 连接进行数据传输，无需每次发送数据都重新建立连接。

•低延迟：适合实时应用，如物联网（IoT）、在线协作、股票交易等。

2.WebSocket 的工作原理

WebSocket 采用 ws:// 或 wss://（安全版，基于 TLS 加密） 协议，建立握手后，客户端和服务器可以随时通信。

连接过程

（1）客户端发起 WebSocket 连接（使用 ws:// 或 wss://）。

（2）服务器响应并完成 WebSocket 握手（HTTP 101 Switching Protocols）。

（3）建立 WebSocket 连接后，双向通信可以随时进行。

（4）任何一方都可以关闭连接。

Babel 是如何工作的1. 解析（Parsing） Babel 首先将源代码解析成抽象语法树（AST）。解析过程分为两个阶段： 词法分析（Lexical Analysis）：将代码字符串分解成一系列标记（tokens），如关键字、标识符、运算符等。 语法分析（Syntactic Analysis）：根据语法规则将 tokens 转换为 AST，表示代码的结构。 2. 转换（Transformation） Babel 遍历 AST，并根据配置的插件或预设（presets）对 AST 进行修改。每个插件通常针对特定的语法特性（如箭头函数、类、模板字符串等）进行转换。例如： 将 const 转换为 var。 将箭头函数转换为普通函数。 将 class 转换为基于原型的语法。 3. 生成（Code Generation） 转换后的 AST 会被重新生成为目标代码。Babel 会遍历 AST 的每个节点，并根据节点类型生成对应的代码字符串。
